Minimum viable documentation for APIs

Minimum viable documentation set (MVD) is the smallest amount of documentation that would allow developers to adopt an API (or other software) with little or no frustration and with the least possible amount of wasted time.

When frustration with docs is high, it's because information is missing, incomplete, or too hard to find. For example, there are no code samples, or no tutorials, or no information about how the API works on a high level.

The MVD includes the most critical information that users need and organizes it in a way that makes things easy to find and use.

Here are the three document types that you need in order to have minimally viable documentation for an API:

  • Reference. For REST APIs, this means complete catalog of endpoints, their methods, the parameters that can be used for the request, and the parameters that can be returned in the response. For the reference section completeness and accuracy are critical. Nothing infuriates users like a parameter with no description at all.

  • A quickstart or Getting Started document.

  • An overview that explains what the API is, how it works, what problem it solves, and any other high-level information the user might want to understand before adopting the API.

There are good reasons for this three-part structure.

Different kinds of users

Some APIs offer only reference information. This turns potential users away and causes unnecessary suffering for some of the developers who end up using the API.

Developers are divided between those who go straight to work with code (sometimes copy-pasting the examples and modifying them) and those who first try to understand how the API works. You must accommodate both. And among those who dig into the code, there are developers who follow a tutorial and developers who go straight to the examples, copy-paste them, and then modify the code for their own purposes.

API users have different ways of engaging with docs: But each type of documentation has a group of users who will be dissatisfied if it is missing or inadequate. So the lack of a quickstart or an Overview will stop numerous developers from ever adopting an API if they can help it. Or, if they are forced by circumstances to use the API, they'll be miserable and might complain to your support department.

The lesson: When documenting an API, you can't omit any of the three types of information. You have to provide documentation for every kind of user.

Different kinds of information

As it turns out, these three kinds of documents are the things developers have come to expect when they adopt new APIs, and that's because there's a larger pattern at work.

The three types of software documentation belong to three general types of information that are natural to all kinds of technical and non-technical writing:

Reference information

Any dictionary-like collection of consistently structured entries, usually consulted when the reader wants to look up just one term as opposed to reading all the way through the section.

Task information

A set of steps that tells the reader how to complete a task when the reader's goal is to get something done. In the MVD, this would be the quickstart.

Conceptual information

A more open form used to explain ideas. Conceptual information isn't as structured as the other two types, and its goal is understanding. In the MVD, the Overview contains most of the conceptual information.

Some people are wary of these information types, either because they're unfamiliar or because they're associated with DITA, and some people don't want to use DITA because it sounds complicated.

To help people get past that, I like to tell the story about my first encounter with these information types. I took a course in software technical writing at UC Berkeley Extension in 2016 or so, and in the first or second meeting the professor who taught that course (Joanna Bujes, now my manager at Synopsys Software Integrity Group) walked around the small classroom with an armload of books, from her own bookshelves, and placed a volume in front of each student. When she finished, each of us was looking at a different cookbook.

I don't remember which cookbook I received, but Professor Bujes pointed out that each book used the information types above:

  • Every cookbook contains not only step-by-step instructions for cooking numerous recipes

  • Reference sections. For example, a table correlating pounds of fruit to the number of quart jars needed to preserve them

  • Conceptual information that explains things such as how to select and store meat, fish, produce and other ingredients; what kind of equipment to use for baking bread, and how to handle dough.

The cookbook is a good analogy for technical documentation, because it demonstrates that the typology that divides material into task/concept/reference is an organic and natural way to organize information. Separating information into these three types of documents makes it easier to find the information you want when you want it.

In other words, readers most likely use reference, task, and conceptual information at different times and for different purposes. Keeping them separate means not only that readers know where to look for the information that they want, but that they don't wade through unnecessary information, unrelated to the task at hand.

For example, a significant number of users might want to read the nine-page explanatory article that begins the bread chapter in The Joy of Cooking -- it explains how to mix bread dough; how to knead and proof correctly; how to achieve hard, soft, or glossy crust; and how to shape dough into loaves. Advanced users might also consult the five-and-a-half page reference section that runs down numerous kinds of flour (The following are just the wheat flours: all-purpose, cake, bread, presifted, pastry, instant, self-rising, semolina, farina, whole-grain, bran, cracked wheat, gluten flour, wheat germ flour, triticale flour. Then there's an entire listing of non-wheat flours.)

Imaging trying to bake bread if the author had to inserted all that information into each of the recipes, most of which run a half page or less.

Readers would most likely read the general bread-making article just once, and early on. They would turn to the step-by-step instructions many times, maybe every single time that they make bread or biscuits. Advanced users might consult the flour reference for special projects. The three types of information are presented separately because it makes sense for the user.

For the same reason, we avoid mixing conceptual information, reference, and tasks when we write software documentation.

This explains why all three document types are necessary. The next logical question is what are the requirements for each of the three types. What must a reference section contain to do its job adequately? What goes into the overview? What is the correct way to write a task? Continue reading for more about those questions.
